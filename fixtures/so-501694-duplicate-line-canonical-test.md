## What the bug looks like

28 lines total on the SO. Only 16 are distinct. 12 are duplicates.

**Three write batches, in DB order:**

1. **Write 1 (22:10:57-59) — initial SO creation from estimate.** 6 lines, all singletons. Clean. (Client notes, W-SERV-Level-1, mainspring 311, GAS, 2day-OBW, Memo-1.)

2. **Write 2 (22:23:33) — bulk insert from user editing.** 10 lines, same timestamp (batch insert). Includes one in-batch duplicate: `2130-540` at sort_orders 9 AND 11. Other lines added: CW-ROL-6dc1cb23, Service Notes, W-watchmaker-EzequielG, 2130-5100., 2135-668, 2130-245., B29-206-1-C5, Recut Fluted bezel 36mm, Bezel Laser Welding.

3. **Write 3 (22:39:25, ~16 min later) — ALL 10 rows from Write 2 re-inserted.** Same parts, same sort_orders, new UUIDs. So `2130-540` ends up with 4 copies total (2 from Write 2 + 2 from Write 3). All other Write-2 lines end up with 2 copies. Write-1 lines unaffected.

## Which bug produced what

- **Write 2's in-batch dup of `2130-540`** = `handleQuickBarcodeSubmit` re-entry (Bug 1 / Fix B). User mashed Enter on the quick-barcode input; async handler ran twice in parallel; both invocations appended the same part.
- **Write 3's complete re-insert of Write 2** = persistOrder UUID patch-back failure (Bug 2 / Fix A). User navigated away (likely to Performance Report), came back, save fired with stale `String(Date.now())` ids in `lineItems`. `existingLineMap` didn't match, all rows fell through to `linesToInsert`.

Session replay corroboration: user entered Part 206C, Recut Fluted bezel 36mm, and Bezel Laser Welding twice each in succession during the editing session. Each appears as 2 copies in Write 2.

## How to use it

**For Fix A validation:**
- After Fix A is applied: reproduce the scenario (open a draft SO, add a line via quick-barcode, save, navigate to Performance Report, return, save again, refresh).
- Expected: one copy of the line, not two. SO 501694's Write 3 pattern should be unreproducible.

**For Fix B validation:**
- After Fix B is applied: in a draft SO with quick-barcode input focused, deliberately mash Enter twice rapidly.
- Expected: only one line added, not two. SO 501694's `2130-540` Write 2 pattern (sort 9 + sort 11) should be unreproducible.

## DB query to inspect

```sql
SELECT id, sort_order, entered_part_number, qty_ordered, unit_price, extended_price, created_at
FROM public.so_lines
WHERE so_id = (SELECT id FROM public.sales_orders WHERE so_number = '501694')
ORDER BY created_at, sort_order;
```

Expect to see the three time-clusters above. The 2130-540 quadruple (4 rows with identical part_number, qty, unit_price) is the most obvious tell.
